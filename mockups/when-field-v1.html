WhenField — one date/time standard

Today two grammars exist. Events use a clean date field + amber scroll-list time picker; multi-day bookings use a cramped native date│time box that clips the date and shows AM/PM (right). The proposal: one WhenField primitive every form uses — single instant, single-day range, or a multi-day span — always our own tap-to-open fields (never a squeezed native box), always at the top of the form, always cleared by the keyboard.

The standard, in one place

  • One primitive, three modes. Every form's date/time flows through WhenField — never a raw native control in a row.
    point — a day + optional time (restaurant, generic event)
    range — same-day start→end (today's event picker)
    span — start→end across any number of days (flight, hotel)
  • Date = the native field, like events. A full-width <input type="date"> (a real OS calendar) that can't be clipped — beside a compact custom time field, so the date always keeps its room. The cropped-date bug in the screenshot came from a wide native time input squeezing the date; a compact custom time field structurally prevents it.
  • Time = the event grammar everywhere. The same amber scroll-list picker (with an exact native fallback, always 24h) drives single-day and span alike, and auto-closes on pick. Span just isn't capped at the overnight cutoff — the end carries a +N day badge and a derived duration read-out.
  • Placement is part of the standard. "When" always sits first in the form body, directly under the identity row (title, or route origin→destination).
  • Keyboard-safe by construction. Picking is done in the panel (no keyboard); the exact-entry input scrolls itself above the keyboard. This lives in the shared form container, so every current and future form inherits it.
