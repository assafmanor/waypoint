# Session 69 — Loading states built (ADR-0105): light boot + content-shaped snapshot skeleton

Built the two ungated pieces of [ADR-0105](../decisions/0105-loading-states-design.md) (designed session 67, [`mockups/loading-states-v1.html`](../../mockups/loading-states-v1.html)): the light "board power-on" boot and the per-mode content-shaped snapshot skeleton. The determinate upload row and the dark boot stay gated on their own prerequisites (below).

## What shipped

- **`ui/feedback/BootScreen`** — replaces the bare-word `BootScreen` (local to `App.tsx`) at all six call sites (auth loading, the anon/login gate, the pending-intent gate, `AppRoutes`' `Suspense` fallback, `RootSurface`'s trips-list load). Cool-paper (`--screen`) ground, an ink mono clock (device-local time — no trip is loaded yet to know its timezone), an amber halo that **ramps** via `.fb-boot-mark::before` (not the reserved `wp-board-pulse` live blip), and an indeterminate departure-line sweep. Every color reads a token via `color-mix`, so the block is remap-ready for the dark boot without any dark-specific rule existing yet.
- **`ui/feedback/HomeSkeleton` + `ui/feedback/ChromeSkeleton`** — the `snapshot.loading` skeleton in `state/trip-state.tsx`'s `TripProvider`, replacing the old two-bar generic shimmer. Two per-mode shapes (Trip: board hero + quick-access tiles + glance card; Plan: prep hero + readiness bar + checklist rows), plus a mode-themed header bar behind it.
- **Mode-at-loading-time (ADR-0105's "chrome is already mode-themed")**: `TripProvider` takes an optional `knownTrip` prop; `RootSurface` already has the full resolved `Trip` (from `loadTripList`) before mounting `TripProvider`, so it's passed straight through — no extra fetch. `deriveMode`/`lib/mode.ts` (existing) picks Trip vs. Plan from it. The header bar shows the _real_ trip name/icon immediately (from `knownTrip`) — only the avatar shimmers, since member data isn't in yet. Falls back to the Trip shape + a placeholder name when `knownTrip` isn't available (e.g. a direct `/trip/:id/settings` deep link).

## A rework worth recording: skeletons must render through the real classes, not a copy of them

First pass built `HomeSkeleton` with its own parallel CSS (`.fb-skel-board`, `.fb-skel-prep`, `.fb-skel-quick`, …) in `feedback.css`, hand-tuned by reading `board.css`/`screens.css` and copying padding/radius/gradient values across. It worked and looked right, but it was exactly the anti-pattern CLAUDE.md's reuse-first rule (ADR-0096) and the ADR-0078/0079/0094/0095 cleanups exist to prevent: a second surface that has to be kept in sync by hand, silently drifting the moment someone resizes `.wp-board`'s padding or `.qa`'s tile height without knowing a skeleton also needs updating.

Caught via a live-app screenshot review (the hero read shorter than the real board, and the header bar was simply missing — `AppShell`'s `header` slot wasn't wired at all in the loading branch). Fixed by rewriting `HomeSkeleton`/`ChromeSkeleton` to render the **real** classes as plain wrapper elements (`.wp-board`, `.wp-board-top`, `.wp-board-now-title`, `.wp-board-divider`, `.wp-board-next-row`, `.wp-board-countdown`, `.wp-board-progress`; `.quick`/`.qa`; `.glance-day`; `.prep`, `.prep-count`, `.prep-ready`, `.prep-track`; `.checklist`, `.chk-row`, `.chk-ic`, `.chk-main`; `.header`/`.mode-chrome`/`.trip-row`/`.trip-name-*`), with `Skeleton` bars as children standing in for text — never merged onto the same element as a real class, to avoid a same-specificity background/color fight between `.fb-skel` and the real rule. `feedback.css` now only adds what the real stylesheets don't already provide: recoloring the shimmer for the two hero surfaces' always-dark/violet backgrounds (`.wp-board .fb-skel`, `.prep .fb-skel`) and the header's Trip-mode (indigo) background, plus one pill-radius utility for the thin progress/readiness tracks. Both real stylesheets (`board.css` via `Home.tsx`, `screens.css` via `App.tsx`) are already eager-bundled, so no new import was needed.

Net effect: a future change to the real board/tile/checklist CSS is inherited by the skeleton automatically, the same way `ChromeSkeleton` already inherited the header's mode theming for free.

## A second pass: abstract, not line-for-line

First cut of `HomeSkeleton` drew a separate thin bar for every real text line (a label line, then a title block, then a meta line, then a next-label line, then a next-title line, …) — busier than it needed to be, and the glance card in particular was much shorter than the real `GlanceCard` (one 26px filler block standing in for a rail + a 32px lead count + a bordered foot line that are, together, closer to 130px of real content). Simplified both: fewer, chunkier skeleton shapes per hero (drop the small standalone label lines, keep one title-weight block + one meta line), and the glance card now mirrors the real `.rail`/`.lead`/`.glance-foot` rhythm — one shimmering bar for the whole rail, one larger block for the lead count, one line under the real border for the foot — so it's abstract (not a rectangle per micro-label) while still landing at the real card's actual height.

## Verified in the real app

Ran the backend + frontend with `DEV_AUTH=1` (system Postgres via `pg_ctlcluster`, no Docker daemon available in this sandbox) and drove it headless with Playwright/Chromium:

- Boot screen renders at every gate; clock ticks; RTL is correct (Hebrew label right-to-left, clock/dots `dir="ltr"`).
- `prefers-reduced-motion: reduce` freezes the glow/dots/sweep at a legible resting position (each animated property has a real, non-off-screen resting value declared outside the keyframes, so the existing app-wide `* { animation: none !important }` rule doesn't need its own override to look right) — confirmed by screenshot.
- Snapshot skeleton captured mid-load (delayed the `/snapshot` fetch via an in-page `fetch` patch, since Playwright's `page.route` interception stalled requests in this sandbox — reason not chased, worked around instead) shows the real header bar with the real trip name, the taller board hero, the quick-access tiles, and the glance card, settling into the identical real layout once the snapshot resolves.
- Plan mode's shape (`.prep` + `.checklist` rows) verified via `HomeSkeleton.test.tsx`/component render — the pre-trip flow needed to reach it live hit an unrelated `401` when the dev clock was pinned via `localStorage` before auth resolved; not chased since it's orthogonal to this change and the component-level coverage plus the shared-mechanism proof from Trip mode were enough.

## Deferred / gated (unchanged from the design)

- **Dark boot** — `BootScreen` reads tokens (`var(--screen)`, `var(--amber)` via `color-mix`) so the remap will apply once `data-theme='dark'` ships (U-08), but no dark-specific rule was added this session.
- **Determinate upload row** — checked: `lib/outbox.ts` already has an `UPLOAD_DOCUMENT` outbox verb (ADR-0056's outbox mechanism is wired), but `lib/api.ts`'s `uploadDocument` reports no progress events yet. `DocumentsSection.tsx`'s row stays the indeterminate `מעלה…` spinner until progress reporting lands.
