# 0035 — In-app "back" is one guarded history step; the PWA return gesture triggers it

**Status:** Accepted — **mechanism superseded by [0090](0090-back-is-computed-from-nav-state.md)** (2026-07-20). The **behavior** below stands and is still the reference — the layer-peeling precedence (§2), the two-tap trip-exit confirm, the single-source day model (§4), and the RTL return gesture (§5). What changed: back is no longer resolved by traversing browser history (`history.back()` / `navigate(-1)` + the `homeBehind` / history-index machinery in the 2026-07-19 refinements); it is computed as a pure function of the current nav state and executed as an explicit navigation. Read §2/§4/§5 for _what_ back does; read [0090](0090-back-is-computed-from-nav-state.md) for _how_ it's now resolved.
**Date:** 2026-07-14
**Refines:** [0024](0024-app-shell-and-trip-lifecycle.md) (app shell & routing), [0033](0033-all-trips-home.md) (all-trips as the "out and across" surface), [0007](0007-platform-pwa.md) (installed PWA), [0017](0017-mobile-first-device-targets.md) (phone-first, touch-first)

## Context

The app is an **installed, `display: standalone` PWA** (ADR-0007). On an installed iOS PWA there is **no browser chrome and no system back gesture at all** — a user who drills in has no way back except affordances we build. On Android/desktop a system back exists, but today it does the wrong thing.

The reason it does the wrong thing: only _some_ of Waypoint's navigation is real browser history. React-router owns the full-page routes (`/login`, `/trips`, `/new`, `/join/:token`, `/trip/:id/settings`, and `/` = the in-trip shell). But **everything _inside_ a trip is React `useState`, invisible to history**: the bottom-nav **tab** (home / map / index / days), the open **overlays** (account sheet, the hard-edit confirm dialog, the Plan-builder gap-fill / event-menu / schedule sheets). So a "back" from deep inside a trip either does nothing (no in-app back exists) or — where a system back does exist — jumps straight out to `/trips`, discarding every in-trip step and every open sheet underneath it.

We want a **return gesture** (edge pull) that "navigates within the app, not the browser's back," _and_ a coherent back model behind it.

## Decision

**1. React-router stays the single owner of history. In-app navigation _becomes_ history; we do not build a parallel gesture-only navigation stack.** The counter-intuitive move: rather than intercept/suppress the browser's back, we make each in-app step a real history entry so that _any_ "back" trigger — Android/desktop system back **and** our own edge gesture — resolves to the same one action. (Suppressing the native back is rejected below.)

**2. There is exactly one back action: `goBack()`**, a guarded `history` step. Precedence — one `goBack()` peels the topmost layer:

1. **An open overlay** (sheet / dialog / picker) → close it.
2. **A non-Home tab** inside a trip → return to **Home** (the anchor).
3. **Home base** inside a trip → out to **`/trips`** (the all-trips "out and across" surface, ADR-0033).
4. **A shell route** (`/new`, `/join/:token`, `/trip/:id/settings`) → its parent.
5. **At `/trips` / zero-state** → **no-op**. Back never falls out to `/login` or exits the app unexpectedly.

**3. Tabs use the Home-anchor model (Material bottom-nav rule).** Home is the base. Home → a non-Home tab **pushes** one entry; switching between non-Home tabs **replaces** (the stack never accumulates a tab trail); so back from _any_ tab returns to Home, and a second back exits to `/trips`. Tab lives in the URL (`?tab=`), so it is a real, reload-surviving history entry. (Rejected: retracing the exact tab-visit order — an unbounded, unpredictable stack.)

**4. Overlays are tracked in a back-consulted stack; `goBack()` closes the topmost before touching structural history.** Sheets/dialogs/pickers register themselves (once, in the `Sheet` and confirm primitives) into an in-memory overlay stack; the first thing `goBack()` does is close the top of that stack if it is non-empty. This is what makes the return gesture close the topmost sheet instead of navigating out from under it, and it needs no per-sheet call-site work. **Structural** navigation (tabs, routes) stays genuine react-router history, so the platform's own back button peels _those_ correctly too. Reconciling the platform system-back with a still-open _transient_ sheet is deferred (§ Consequences) — it is moot on installed iOS, the motivating target, which has no system back. Day selection on the header strip is deliberately **not** a back layer — it is a lateral view change within the Days tab, not a drill-in, so back does not walk through every day you tapped.

**5. The return gesture is one _trigger_ of `goBack()`, not its own logic.** A trailing-edge horizontal pull calls `goBack()`. Because the app is **full RTL** (ADR-0009), the platform back convention mirrors: the activation edge is the **right** edge and the pull is **toward the left** (the `‹` back chevron itself flips to `›`). The gesture reads `dir`, it does not hard-code a side. It activates only from a narrow (~24px) trailing-edge zone, so it does not fight the horizontally-scrolling day-strip or the Plan-builder's pointer-capture drag-to-reorder; it shows a live peek/parallax of the surface behind and commits past a distance **or** velocity threshold, else snaps back. It is back-only (no forward/redo on a mobile pull).

**6. Root guard.** At the in-trip Home base, `goBack()` navigates **explicitly** to `/trips` rather than stepping blindly into whatever preceded the trip — so a trip opened from a cold deep link still lands somewhere sensible instead of exiting the app. At `/trips` and the zero-state, `goBack()` is a no-op.

## Consequences

- **Coherence for free on Android/desktop too:** because tabs and routes are now genuine history entries, the hardware/browser back button already peels _those_ correctly — the custom gesture's unique jobs are (a) to exist on installed iOS where no system back does, (b) the root guard, and (c) closing an open overlay (which system back does not yet do — see deferred).
- **Reload/deep-link friendliness:** `?tab=` means a refresh keeps your tab; the in-trip surface is now URL-addressable.
- **The primitives carry the wiring:** `Sheet` and the confirm provider register as overlays, so every current sheet (account, gap-fill, event-menu, schedule) and future ones inherit back-to-close without per-call-site work.
- **Two invariants keep this working as the app grows** (stated for contributors in [`engineering/conventions.md`](../engineering/conventions.md) → Frontend, and one now lint-guarded): (a) **overlays render through the `Modal` primitive** so they auto-register — a bespoke `createPortal`/`fixed` panel that skips `useOverlay` is invisible to back, so `eslint.config.mjs` restricts `createPortal` to an allowlist (`Modal` + the already-back-aware `DocumentViewer`); (b) **inside the trip shell only Home→tab may _push_ history; every lateral/in-place change uses `replace`** (a lateral day change via `daySelectTarget` does), so the Home anchor stays exactly one entry below any tab. The root-guard surfaces are single-sourced as `ROOT_PATHS` in `state/nav-state.tsx`, mirroring `AppRoutes`, so adding a top-level surface is one obvious edit.
- **Bounded history growth:** stepping Home→trip repeatedly can add `/trips` entries (root guard pushes rather than pops). Acceptable at this app's scale; can be optimized to a pop-when-parent later.
- **Deferred (recorded, not built):** a visible header back-chevron mirroring the gesture on shell routes; a forward gesture; per-day history. (Platform **system-back** closing overlays / confirming trip-exit is now handled on Android via the Navigation API — see Refinements 2026-07-15b.)

## Refinements

**2026-07-15 (Assaf, post-merge feedback).** Three adjustments — additive, the decision above stands:

1. **Leaving a trip is confirmed.** Back at the in-trip **Home base** no longer exits straight to `/trips`; the first back **arms** a toast ("עוד פעם ויוצאים מהטיול 👋") and a second back within ~3s actually leaves (the Android "press again to exit" pattern). Guards against an accidental swipe yanking you out of the trip you're using on the ground. Surfaced as a distinct `exit-trip` step so the confirm lives in the handler, not the pure decision.
2. **Overlays dismiss from anywhere.** The edge-zone restriction is lifted **while a sheet/dialog is open** — over a modal there's nothing to scroll under, so a back pull can start mid-screen. Makes the account sheet (and every `Sheet`/dialog) reliably closeable by the return gesture, not only by an exact edge swipe.
3. **Native slide, not a nudged pill.** The peek chevron is gone; the screen now tracks the finger ~1:1 (rubber-band past halfway) with a trailing shadow, and a committed **structural** back slides the screen fully off before the content swaps. Overlay-dismiss and the exit-confirm spring back instead of sliding.

**2026-07-15b (Assaf, Android).** The above rode the custom edge gesture — but **on Android the OS owns the screen edges** (system back gesture), so it pre-empts the custom swipe, and the system back button/gesture calls the browser's `history.back()`, which bypassed the overlay-close and leave-trip confirm (only tabs/routes, being real history, responded). Fix: intercept the back **traversal at its source** with the **Navigation API** (`navigation` `'navigate'` event, Chromium/Android; absent on Safari/iOS where there's no system back to reconcile). On a backward traverse we `preventDefault()` and run the same intent — `close-overlay`, `arm-exit`, `do-exit` — or `allow` it (no cancel) so react-router's own back peels the tab/route. No history guards: the interception happens before react-router sees the pop, so there's still one history owner. The slide **animation** stays gesture-only — a hardware back is instant by nature. Pure `systemBackDecision()` is unit-tested; see `state/nav-state.tsx`.

**2026-07-18 (Assaf, triage).** Two adjustments to how back lands on Home; the §2 precedence stands.

1. **Back to Home resets the day to today (Trip mode).** Today the day-strip snaps to today **only on the nav-bar Home-tab _tap_** (`App.tsx Shell.onSelectTab:351-354`); the `goBack()` / return-gesture path to Home (`useTripTab` `goToTab('home')`, `nav-state.tsx:278-287`) does **not** reset `activeDate`. So swiping back from a non-Home tab lands you on Home but still on whatever day you'd navigated to — which doesn't feel like "returning to the trip's home." Assaf: "צריך לחזור לתצוגת בית טיול (והיום של הטיול)." Fix: the **structural back-to-Home step also resets `activeDate` to today in Trip mode**, so any route to Home (tap _or_ gesture) lands on Home _and_ today. **Plan mode preserves the selected day** (unchanged — Plan isn't today-anchored).

_As built:_ the reset lives at the single point every route to Home converges on — the active tab becoming Home — rather than in each caller. An effect in `App.tsx Shell` keyed on the tab fires `setActiveDate(today)` when `shouldResetDayToToday(tab, mode)` (`nav-state.tsx`) holds, so the nav-bar tap, `goToTab`, the return gesture and system-back all reset uniformly. This supersedes the earlier per-tap reset at `App.tsx Shell.onSelectTab`.

2. **Which was the reported bug.** Assaf reported back "not returning to the trip home/today." The concrete cause is exactly #1 — the gesture path skipping the day reset — so the fix above resolves it; there is no second defect in the §2 precedence (`structuralBackStep`/`systemBackDecision` are well-formed and unit-tested). The implementer should still verify end-to-end that a non-Home tab → Home and Home → `/trips` (behind the two-tap exit confirm, refinement 2026-07-15) both resolve correctly via the gesture; record any further failing case here if found. This composes with ADR-0060 (idle-resume also lands on Home + today) — both routes converge on "Home shows today."

**2026-07-19 (Assaf, triage).** One hardening to §3; the tab model stands.

**The nav-bar Home tab could silently do nothing** while the other tabs (day / index / map) worked. The Home tab is the only one that navigates by a history `back` (`navigate(-1)`) rather than to an explicit `?tab=` URL — the anchor model's whole point (§3). `tabStep` chose `back` vs. an explicit `replace('/')` on `historyIdx() > 0`, i.e. "does _any_ history exist behind me." But that is **not** the same as "is the trip **Home base** the entry directly behind me." When foreign entries sit behind a tab — the Google OAuth round-trip (ADR-0004/auth-and-google), an external / installed-PWA launch, a shared-link cold open, or a react-router `idx` desync — `historyIdx() > 0` is true yet a blind `back` traverses to something that isn't Home; if that target is cross-origin or immediately redirects forward, the tap resolves to **no visible change** ("unclickable"). The URL-navigating tabs were immune because they never depend on what's behind them. (This matches "in some conditions I can't retrace" — the trigger is _how the app was launched_, not an in-app sequence.)

_Fix:_ take the `back` shortcut only when Home is **provably** the entry directly behind. A Home→tab push sits exactly one entry ahead of the base and tab→tab replaces never advance past it, so we record the Home base's history index whenever we're sitting on it (`homeBaseIdxRef` in the nav context, set by a `tab === HOME_TAB` effect in `useTripTab`) and gate `back` on `historyIdx() === homeBaseIdx + 1`. Otherwise — a cold deep link onto a tab, or foreign / desynced history — route to `/` explicitly. `tabStep`'s third argument changes meaning from `canGoBack` to `homeBehind` (same `back` / `replace('/')` outcomes, now driven by the correct signal). Normal in-app flows are byte-for-byte unchanged (Home is always one entry behind, so `homeBehind` holds and `back` is still taken); only the previously-broken foreign-history cases now land deterministically on Home. Pure `tabStep` stays unit-tested (`state/nav-state.test.ts`). The return-gesture / system-back path to Home (`structuralBackStep`) shares the `back`-from-a-tab shortcut and the same assumption; it was not reported failing and is left as-is — revisit here if a foreign-history case surfaces on the gesture too.

**2026-07-19b (Assaf, follow-up).** The foreign-history case _did_ surface on the other two back triggers, plus a related root-guard leak — the same root cause as 2026-07-19, now closed on **every** path so all three converge on the corrected `homeBehind` signal.

1. **The return gesture (swipe) stranded intermittently.** `structuralBackStep` — the gesture's and desktop system-back's decision — still chose `back` vs. `replace('/')` from a non-Home tab on the old `canGoBack` (`historyIdx() > 0`) signal, so a swipe-back from a tab with foreign history behind (OAuth round-trip, installed-PWA cold launch, `idx` desync) traversed into that foreign entry instead of Home — "sometimes working, sometimes not," keyed on _how the app was launched_. _Fix:_ `structuralBackStep` now takes `homeBehind` and gates the non-Home-tab `back` on it exactly like `tabStep`; when Home isn't provably behind it routes to `/` explicitly. Both call sites share one `homeIsBehind(homeBaseIdxRef.current)` helper so the tap and the gesture can never diverge.

2. **On Android the swipe _is_ the system back**, so it runs through the Navigation API interceptor (`systemBackDecision`), not the custom gesture. That path resolved a non-Home tab to `allow`, letting the OS traverse into foreign history — the Android twin of #1. _Fix:_ `systemBackDecision` gains the same `homeBehind` input and a new `go-home` decision: on a non-Home tab it `allow`s the native back only when Home is provably behind, else cancels and navigates `/` explicitly.

3. **Back at the in-trip Home base exited the app instead of landing on `/trips`.** The interceptor early-returned whenever the destination index wasn't a smaller number — but when the Home base sits at history index 0 (a cold launch / OAuth round-trip straight into a trip), the OS back leaves the app and its destination index is indeterminate, so the guard skipped the `arm-exit` / `do-exit` confirm and the app just closed. _Fix:_ the guard now skips only a **provably-forward** traverse; a backward or app-leaving one still routes through the intent, so the root guard `preventDefault()`s and keeps the user in-app (arm → `/trips`) whenever the traversal is cancelable. (When the OS reports the traversal non-cancelable there is nothing a page can do — unchanged, platform-bound.)

All four back triggers — the nav-bar Home tap, the return gesture, desktop system-back, and Android system-back — now resolve Home and the trip-exit through the same `homeBehind` / root-guard logic. Pure `structuralBackStep` and `systemBackDecision` stay unit-tested (`state/nav-state.test.ts`).

**2026-07-19c (Assaf, triage). The selected day gets one source of truth; this supersedes the 2026-07-18 reset-effect (§4 day model).** The day-strip developed a "needs a double-tap" bug: selecting a day did not move to the day view, and tapping Home did not reliably land on Home+today, in **both** modes.

_Cause:_ the selected day had **two** copies — `activeDate` (React state) and `?day=` (URL) — kept in sync by a mirror effect (state→URL) and an adopt effect (URL→state), _plus_ a third effect that reset the day to today on the Home tab (2026-07-18). On landing on Home these raced: the reset set `activeDate = today` while the mirror, still seeing the stale day in the same flush, wrote `?day=<stale>` back, which the adopt effect then re-applied — a ping-pong that settled on the wrong day and needed a second tap to correct. It was **not** the back-decision (`structuralBackStep`/`systemBackDecision`) — those were unchanged.

_Fix — single source of truth:_ the selected day lives **only** in `?day=`. `activeDate` is now **derived** each render (`state/trip-state.tsx`), tab-aware: on the **Home tab it is always today**, in **every mode** (a stray `?day=` on a Home URL cannot override it); off Home it resolves from `?day=` (clamped to the reactive trip range, invalid → today). Selecting a day goes through one pure step, `daySelectTarget` (`state/nav-state.tsx`, unit-tested): it writes `?day=` (omitted when today, so the URL stays clean) **and** lands on the `days` tab in a single navigation, so the day can never be dropped by a follow-up tab move. Because Home is only ever reached without a `?day=`, **"Home shows today" is structural, not an effect anyone can forget to fire** — the mirror, adopt, and reset effects are all deleted. Back/forward and reload/share keep working for free (the URL _is_ the state). This **reverses the 2026-07-18 Plan-mode carve-out**: Home is today-anchored in Plan mode too (Assaf: "both trip mode and plan mode"). ADR-0060's idle-resume no longer needs its own day-snap — `navigate('/')` lands on a `?day=`-free Home, which derives to today.

## Alternatives considered

- **Custom gesture that suppresses the native back.** Rejected: iOS Safari's edge-swipe cannot be reliably prevented, and fighting Android's back yields two desynced back mechanisms. Owning history instead of blocking it is simpler and correct.
- **Raw `history.pushState` traps alongside react-router.** Rejected: two owners of the History API desync react-router's internal index. We express every layer _through_ react-router (`navigate` + `location.state`) so there is one owner.
- **Tabs as peers, back exits immediately from any tab.** Rejected in favor of the Home-anchor rule — jumping out of the trip from a deep tab is abrupt; Home is a predictable anchor.
