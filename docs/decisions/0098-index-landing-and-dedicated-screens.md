# 0098 — Index becomes a bookings/documents landing with dedicated screens, filter/search/collapse, and a motion pass

**Status:** Accepted (Assaf sign-off 2026-07-20)
**Date:** 2026-07-20
**Refines:** [0049](0049-index-tab-mode-and-lifecycle.md) §3 (bookings and documents stay peers with matched add affordances — they now each get their own screen off a shared landing, instead of sharing one page), [0090](0090-back-is-computed-from-nav-state.md) (back-to-landing reuses the existing overlay-stack registration, no new `resolveBack` rule), [0050](0050-home-quick-access-deep-links-and-empty-states.md) (the Home quick-access deep-links into the Index; the landing structure changes how they resolve), [0061](0061-plan-home-readiness-rework.md) (the past-bookings collapse mirrors its completed-checks toggle)
**Relates:** [0004](0004-integrations-are-pipes.md), [0011](0011-hard-soft-event-model.md), [0028](0028-plan-violet-color-budget-dark-ready.md), [0096](0096-per-domain-claude-md-guides.md)

## Context

The Index tab (ADR-0049) renders bookings and documents as two sections on one scrolling page. That reads fine at the sizes it shipped with, but a longer trip (more legs, more lodging, more documents per traveller) makes the single page hard to scan: the two sections compete for the same scroll, and a section with many rows crowds out the other.

Two directions were mocked up side by side:

- `mockups/index-findability-v1.html` — keep one page; add a category filter chip row, a search box, past-bookings collapsed by default (mirroring ADR-0061's Plan Home toggle), and a chevron that shrinks either whole section to one line.
- `mockups/index-findability-split-v1.html` — turn the Index into a small **landing** with two tiles (🎫 bookings, 📎 documents), each showing a count + a one-line preview; tapping a tile pushes a dedicated full screen for that content, with a back control to return.

Assaf also asked about adding more content types later (notes, research, media). Those are **not decided or scoped by this ADR** — the split variant's landing was stress-tested with three disabled placeholder tiles purely to see whether it holds up at 5 tiles instead of 2 (it does — screenshot in the session record), and that scale trajectory is one reason to prefer the landing shape now, over a page that would otherwise grow another whole section per future type. Nothing beyond bookings + documents ships from this ADR; if/when a new content type is actually decided, it gets its own ADR and becomes one more tile on this same landing (see the "מחקר" naming flag below).

## Decision

**1. The Index becomes a landing with two tiles, each pushing a dedicated screen.** A bookings tile (count + "next: <title> · <when>" preview) and a documents tile (count + the present document-type groups as a one-line summary). Tapping a tile shows that content full-screen; a back control returns to the landing. Bookings and documents stay peers (ADR-0049 §3) — matched tile treatment, matched screens — just no longer stacked on one page.

**2. Within the bookings screen:** a category filter chip row (הכל + the 6 `BookingType` values, neutral — ink-filled when selected, never tinted per category, since amber/teal/violet stay reserved per the design-language color budget and a booking's own badge already carries its icon/category); a search control (icon → text input) filtering by title/confirmation code across upcoming and past (a non-empty query force-opens the past list so a match hiding there isn't silently missed); and past bookings collapsed by default, mirroring ADR-0061's Plan Home "completed checks" toggle **exactly** (a count-in-label button, tap to reveal) — reusing an already-accepted interaction rather than inventing a new one. The documents screen keeps its existing type-grouped list, unchanged.

**3. Every directional indicator is the real `ui/NavArrow.tsx` / `ui/Icon.tsx` SVG, never a Unicode arrow/chevron character.** Characters like `‹`/`›`/`→` are `Bidi_Mirrored` and silently flip direction inside `dir="rtl"` — confirmed by a rendering test during this work (an intended-left `‹` rendered pointing right; a route arrow rendered pointing at the origin instead of the destination). This isn't new infrastructure, just consistent use of what already exists: `NavArrow`'s `forward`/`back` variants (forward points left in RTL, back points right, via explicit `scaleX(-1)`, never glyph mirroring) and `Icon`'s `caret`.

**4. Motion, not more color or content, is the "wow" layer.** The color budget and Home's board-hero (ADR-0028, "the board is rationed to Home") stay off-limits, so the polish pass is restrained, native-feeling motion instead: the past-bookings list animates open/shut (max-height + opacity) instead of snapping via `display:none`; a chip/search filter change reveals matching rows with a small per-row stagger instead of an all-at-once flip; every tappable control (tile, chip, search icon, addbtn, past-toggle) gets a brief scale-down on tap. All of it must honor `prefers-reduced-motion`.

**5. Navigation: the bookings/documents screens are local view state inside `Index.tsx`, not new routes — and back-to-landing reuses the existing overlay-stack, not a new `resolveBack` rule.** This is the load-bearing implementation decision, so it's spelled out in full:

- `resolveBack`'s rule 2 (`state/nav-state.tsx`) is unconditional: **"a non-Home tab in a trip → Home."** If the bookings/documents screens were just another `?tab=` value or a sub-param read by that same rule, a back/system-back/edge-gesture press from inside one would jump straight to Home, skipping the landing entirely — a real regression, not a cosmetic one.
- Modeling them as real routes (`/trip/:id/index/bookings`, mirroring `/trip/:id/settings`) was considered and **rejected**: `TripSettingsRoute` mounts a fresh `<TripProvider tripId={id}>`, which is right for a screen reached from outside the trip shell but wasteful and inconsistent here — `Index` already renders inside the one `TripProvider` the trip `Shell` mounts, keyed off `?tab=index` (`App.tsx`'s `Screen` function). A second route would remount trip context for no reason.
- **The correct, already-existing mechanism is `useOverlay(onClose)`** (`state/nav-state.tsx`), the same hook `Modal` calls for every sheet/dialog — and `Index.tsx` already uses indirectly today, via `BookingDetail`/`BookingManageSheet`/`BookingSheet`. `useOverlay` is decoupled from `Modal`'s portal/scrim visuals; it is just "register a close callback while mounted, so `resolveBack`'s `hasOverlay` check (rule 1, which runs **before** rule 2) peels it first." A bookings/documents sub-view component — conditionally mounted only while that view is open, exactly like `BookingDetail` is only mounted while `detail` is set — calls `useOverlay(() => setView('landing'))` in its body. One back/gesture/system-back then closes the sub-view back to the landing; only a second back (once already on the landing) falls through to the normal tab rule. **Zero changes to `state/nav-state.tsx`, no new route, no `TripProvider` remount** — this is exactly the additive shape ADR-0090 promises ("adding an overlay is a `useOverlay()` call, not a new decision branch").
- The existing Home quick-access deep-links (ADR-0050: `?booking=<id>` opens a booking, `?focus=docs` used to `scrollIntoView` the documents section) need a small, direct adaptation, not a redesign: `?booking=<id>` should set the local view to `'bookings'` and open the detail sheet on top of it (the sheet still layers over the bookings screen exactly as before); `?focus=docs` should set the view straight to `'documents'` (there is no longer a docs section to scroll to — it's its own screen).

## Reuse audit (ADR-0096)

Per root `CLAUDE.md` rule 8, before writing anything new:

- **Booking/document rows** — unchanged. Both screens keep using `ListRow`/`RowManageSheet` (`ui/domain`), already the shared managed-list-row shape.
- **Category filter chips — extend `ChoiceGrid` (`ui/primitives/ChoiceGrid.tsx`), don't build a new component.** It's already a controlled, generic `Choice<T>` single-select `radiogroup` with a neutral (`--cta`-only, never a semantic hue) selected state — exactly the filter's shape and exactly its color rule. The only gap is layout: `ChoiceGrid` renders a fixed CSS grid (sized for ~4 form cards); the filter needs a horizontally-scrollable pill row that fits 7 options on a narrow phone. That's a small addition (a `variant`/layout prop + CSS), not a rewrite — the kind of "small extraction" CLAUDE.md says to prefer over a second one-off. If it turns out to need materially more than that once someone is in the code, stop and ask rather than silently forking it.
- **Past-bookings collapse — generalize alongside Plan Home's checklist collapse, don't add a second copy.** `screens/PlanHome.tsx`'s `showCompleted` state + `.chk-toggle`/`.chk-done-sum` (screens.css) is the **existing one-off** this ADR's past-toggle is deliberately modeled after. Building Index's version as its own separate `useState`/CSS pair would be precisely the pattern root `CLAUDE.md` calls out — "several ADRs in this repo (0078, 0079, 0094, 0095) exist only to undo the parallel copies that piled up before someone generalized them; don't create the next one." The dev session should extract a shared collapsible-summary primitive (toggle button with a count-in-label, expand/collapse) that both `PlanHome` and `Index` call, carrying this ADR's animated open/shut motion so Plan Home's collapse gains the same polish rather than staying the one un-animated copy.
- **"No results for this filter" — use `ui/feedback`'s `EmptyState`, not a bespoke div.** A hand-rolled empty shell is the exact anti-pattern ADR-0078 replaced roughly six copies of; this is a seventh candidate if built as one more one-off `<div>`.
- **The landing tile is net-new** (`ui/domain/`, presentational, props-driven) — no existing component fits (`StatTile` is a static value+label, not a tappable navigational card). Two call sites only (bookings, documents); this is the right altitude for a first version — don't design in a "content-type registry" for hypothetical future tiles that aren't decided yet.
- **Navigation** — no new infrastructure at all; see Decision §5. The one new pattern (a conditionally-mounted view component calling `useOverlay`) is additive, following the exact shape `BookingDetail`/`BookingManageSheet` already establish.

## Consequences

- No backend or `@waypoint/shared` change — every field the tiles/screens need (booking/document counts, the "next" booking, document type groups) already exists; this is presentation + local navigation state only.
- New copy in `i18n/he.ts` (tile previews, the back label, the "no results" empty state, the search placeholder) — no hardcoded strings, per convention.
- `mockups/index-findability-split-v1.html` is the accepted design reference; `mockups/index-findability-v1.html` is kept as the superseded single-page exploration (design rationale for why the split was chosen over it), not a build spec.
- The "מחקר" (research) naming collision flagged during exploration — that word already names the shipped Map-tab place-search feature (feature-catalog.md) — is **not resolved here**, because no research tile ships from this ADR. Whoever eventually proposes a research/notes/media tile must resolve that naming question in its own ADR before building it.
- Follow-on: Plan Home's checklist collapse (ADR-0061) gains the animated open/shut motion as a side effect of the shared-primitive generalization above, which is a welcome consistency win, not scope creep — flag it in the PR, don't let it balloon into an unrelated Plan Home redesign.

## Alternatives considered

- **Ship the single-page mockup (`index-findability-v1.html`) instead** — cheaper (no navigation-layer question, no back-stack design), and does solve "two sections crowd one page." Rejected because it doesn't scale as cleanly if the Index ever needs a third content type, and a landing-of-tiles is the more natural shape for "an index" in the literal sense.
- **A sticky bookings/documents jump-nav** (an earlier draft inside the single-page mockup) — superseded quickly by the per-section shrink chevron, then by the split entirely; kept only as design history in the mockup file's own comments.
- **Real routes for the sub-screens** (`/trip/:id/index/bookings`) — rejected; see Decision §5 (needless `TripProvider` remount, inconsistent with how every other tab renders inside the one trip shell).
- **A new `resolveBack` precedence rule for "a tab has an open sub-view"** — rejected in favor of reusing the existing overlay-stack (`useOverlay`), which already does exactly this job with zero changes to the shared back-decision function.
